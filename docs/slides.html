Core Java — Data Types & Precision

⚠ Most common silent bug in hackathons

❌ Wrong

int income = 50000;
int emi = 22000;
// gives 0, not 0.44!
System.out.println(emi / income);

✅ Correct

int income = 50000;
int emi = 22000;
// gives 0.44
System.out.println(
  (double) emi / income);
💡 Use double for money & ratios. Round only the final output.

Core Java — Control Flow

Guard clauses — fail early

if (income <= 0) {
  System.out.println(
    "Invalid: income must be positive");
  return;
}
// safe to proceed

Slab order matters

// CORRECT — high first
if (income >= 1000000)      { ... }
else if (income >= 500000)  { ... }
else if (income >= 250000)  { ... }
// WRONG — low first catches all

Applies to: tax slabs · tariff billing · toll fares · reward tiers

Core Java — Loops & Accumulator

// One pass: min, max, average together
double total = 0;
double min = prices[0], max = prices[0];

for (int i = 0; i < prices.length; i++) {
    total += prices[i];
    if (prices[i] < min) min = prices[i];
    if (prices[i] > max) max = prices[i];
}
double avg = total / prices.length;
💡 One clean pass > multiple loops. Watch off-by-one at slab boundaries.

Method Decomposition

One method = one responsibility

public static void main(String[] args) {
    String input  = readInput();
    if (!validateInput(input)) return;
    double result = computeResult(input);
    printOutput(result);
}
📥 readInput()
Collect raw data
🛡 validateInput()
Guard clauses
⚙️ computeResult()
Business logic
📤 printOutput()
Format & display

OOP — When to Use Classes

❌ Parallel arrays — de-sync nightmare

String[] names    = {"Alice","Bob"};
double[] salaries = {50000, 60000};
// One update forgets the other!

✅ Encapsulated object

class Employee {
  private String name;
  private double basicSalary;

  public double computeNetPay() {
    double hra = basicSalary * 0.4;
    double pf  = basicSalary * 0.12;
    return basicSalary + hra - pf;
  }
}

Use OOP when behavior differs by type or multiple fields travel together.

Collections — The Hidden MVP

NeedUseExample
Ordered itemsList<T>Cart, price ticks, log lines
No duplicatesSet<T>Unique customers, keywords
Key→Value lookupMap<K,V>Word counts, category rates
FIFO queueQueue<T>ATM queue, task scheduler
💡 Choose by behavior, not by habit.
Most hackathon problems need Map<String, Integer> for counting or lookup.

Collections — Map in Action

// Keyword frequency counter (Log Analyzer problem)
Map<String, Integer> freq = new HashMap<>();
String[] tokens = log.split("\\s+");

for (String token : tokens) {
    freq.put(token, freq.getOrDefault(token, 0) + 1);
}

// Print top keywords
freq.entrySet().stream()
    .sorted(Map.Entry.<String,Integer>comparingByValue().reversed())
    .limit(5)
    .forEach(e -> System.out.println(e.getKey() + " : " + e.getValue()));

Applies to: PS-17 Log Keyword Counter · PS-10 Reward Points · PS-18 CRM Dedup

Spring Boot — Layered Architecture

🌐 Client (Browser / Postman)
↓ HTTP Request
🎯 Controller — thin, validates & routes
↓
⚙️ Service — ALL business logic lives here
↓
🗄 Repository — only fetches & saves

Spring Boot — Minimal Skeleton

Controller (thin)

@RestController
@RequestMapping("/api/submissions")
public class SubmissionController {
  @Autowired SubmissionService svc;

  @PostMapping
  public ResponseEntity<?> create(
      @RequestBody Submission s) {
    return ResponseEntity.ok(
      svc.create(s));
  }
}

Service (logic)

@Service
public class SubmissionService {
  @Autowired SubmissionRepository repo;

  public Submission create(Submission s) {
    if (s.getStudentName() == null)
      throw new IllegalArgumentException(
        "Name required");
    s.setStatus("SUBMITTED");
    return repo.save(s);
  }
}

MySQL — Schema Design First

3 questions before writing SQL

1️⃣ What are my entities?
2️⃣ What are their relationships?
3️⃣ What will I query most?
CREATE TABLE submissions (
    id           INT PRIMARY KEY AUTO_INCREMENT,
    student_id   INT NOT NULL,
    project_id   INT NOT NULL,
    submitted_at DATETIME DEFAULT NOW(),
    status       ENUM('SUBMITTED','EVALUATED',
                      'PASSED','FAILED')
                 DEFAULT 'SUBMITTED',
    score        DECIMAL(5,2),
    FOREIGN KEY (student_id) REFERENCES students(id),
    FOREIGN KEY (project_id) REFERENCES projects(id)
);

Integration Blueprint

✅ Happy Path

1. Request arrives at Controller
2. Controller validates required fields
3. Service checks business rules
4. Repository saves to MySQL
5. Return 200 OK + data

❌ Error Paths

Missing field → 400 Bad Request
Rule violated → 422 + message
Not found → 404
DB error → 500

⚡ Design happy path AND failure path together.
